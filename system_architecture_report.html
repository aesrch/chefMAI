System Architecture & Technical Report

A detailed engineering study of Chef MAI's vision pipelines, machine learning matching formulas, and live feedback metrics.

System Overview

Chef MAI is an intelligent kitchen assistant designed to minimize food waste and optimize domestic meal planning. By incorporating computer vision (for item extraction), custom suitability classification (via Naive Bayes), and sequential user interaction profiling (via Bayesian learning), the system recommends recipes tailored both to the physical ingredients available in a user's fridge and their personal dining history.

The Ktor (Kotlin) backend and Python FastAPI CV servers act in symbiosis, serving custom recommendation responses directly to a Vite React client dashboard.

Computer Vision (CLIP Ingredient Detection)

The ingredient classification pipeline is managed by a standalone Python FastAPI service. Rather than utilizing standard bounding-box object detectors (e.g. standard YOLOv8) which struggle with novel or fine-grained ingredient categories without substantial domain datasets, Chef MAI implements an open-vocabulary semantic matching model leveraging OpenAI's CLIP (specifically `ViT-B/32`).

Open-Vocabulary Search Space

The model cross-references incoming visual streams with strings matching three key food categories:

Category Vocabularies Evaluated
Proteins chicken, pork, beef, fish, shrimp, crab, egg, tofu
Vegetables onion, garlic, tomato, carrot, potato, bell pepper, cabbage, lettuce, spinach, broccoli, cauliflower, eggplant, cucumber, zucchini, green beans, corn, peas
Fruits apple, banana, orange, lemon, lime, grapefruit, pear, peach, plum, apricot, mango, pineapple, watermelon, cantaloupe, honeydew, kiwi, cherry, grape, etc.

For an uploaded image, the CLIP model determines semantic affinity scores between the image features and text features representing candidate labels:

$$\mathbf{x}_{\text{image}} \leftarrow \text{Preprocess}(\text{Image})$$ $$\mathbf{t}_i = \text{tokenize}(\text{"a photo of ingredients including } \text{ingredient}_i\text{"}) \quad \forall i \in \{1, \dots, N\}$$ $$P(\text{ingredient}_i \mid \text{image}) = \text{Softmax}(\text{CLIP}(\mathbf{x}_{\text{image}}, \mathbf{t}_1, \dots, \mathbf{t}_N))_i$$ $$\mathcal{D} = \{\text{ingredient}_i \mid P(\text{ingredient}_i \mid \text{image}) > 0.10\}$$

FastAPI handles the model execution. On the backend, Ktor calls this service asynchronously using CIO HttpClient: 

suspend fun detectIngredients(imageBytes: ByteArray): String {
    val response = client.post("$baseUrl/detect") {
        setBody(MultiPartFormDataContent(formData {
            append("file", imageBytes, Headers.build {
                append(HttpHeaders.ContentType, "image/jpeg")
                append(HttpHeaders.ContentDisposition, "filename=image.jpg")
            })
        }))
    }
    return response.bodyAsText()
}

Naive Bayes Suitability Classification

The NaiveBayesService classifies recipes as suitable or unsuitable for a user. By evaluating specific categorical dimensions of a recipe (independent of direct rating counts), it models generalized user compatibility.

Extracted Classification Features

  • Genre Flag: e.g., genre:asian, genre:italian, genre:american.
  • Jaccard Match Score Bucket: match:high (≥ 80%), match:medium (≥ 50%), match:low (≥ 20%), or match:none.
  • Complexity Bucket: ingcount:small (≤ 5), ingcount:medium (≤ 10), or ingcount:large.

Using **Laplace Smoothing** ($\alpha = 1.0$) to calculate features probabilities over a dataset containing $N_{\text{samples}}$:

$$P(Class) = \frac{count(Class) + \alpha}{N_{\text{samples}} + 2\alpha}$$ $$P(feature_j \mid Class) = \frac{count(feature_j, Class) + \alpha}{count(Class) + \alpha \times N_{\text{unique features}}}$$

To avoid decimal underflow, probabilities are summed in log-space:

$$\log P(Class \mid \mathbf{f}) = \log P(Class) + \sum_{j=1}^{3} \log P(f_j \mid Class)$$

The probabilities are subsequently converted back into standard scale using the **log-sum-exp subtraction trick** to prevent numerical instability:

$$P(suitable \mid \mathbf{f}) = \frac{e^{\log P(suitable \mid \mathbf{f}) - M}}{e^{\log P(suitable \mid \mathbf{f}) - M} + e^{\log P(unsuitable \mid \mathbf{f}) - M}}$$ $$\text{where } M = \max(\log P(suitable \mid \mathbf{f}), \log P(unsuitable \mid \mathbf{f}))$$

Bayesian User Preference Learning

Rather than employing static profile parameters, Chef MAI uses a dynamic **Beta-Binomial conjugate model** representing the probability $\theta$ that a user will enjoy a given recipe:

$$\theta \sim \text{Beta}(\alpha, \beta)$$ $$\text{Prior parameters: } \alpha_{\text{prior}} = 1.0, \quad \beta_{\text{prior}} = 1.0 \quad \text{(Uniform Prior)}$$

When a user interacts with the app, the posterior parameters ($\alpha, \beta$) are updated based on positive (success) or negative (failure) interaction logs:

User Action Weight Weight ($W_i$) Statistical Meaning
Cooked +3.0 Strong positive signal (adds to $\alpha$)
Rated +2.0 × (Stars / 5.0) Scale-adjusted success signal (adds to $\alpha$)
Liked +2.0 Moderate positive signal (adds to $\alpha$)
Saved +1.5 Mild positive signal (adds to $\alpha$)
Viewed +0.5 Informational context signal (adds to $\alpha$)
Disliked -2.0 Explicit negative signal (adds $|W_i|$ to $\beta$)

Belief Revision Equations: For a target recipe $r$ with genre $G$:

$$\alpha_{\text{post}} = 1.0 + (2.0 \times \text{genreAffinity}) + \sum_{i \in \text{Recipe } r} W_i^+ + \sum_{j \in \text{Genre } G} (0.3 \times W_j^+)$$ $$\beta_{\text{post}} = 1.0 + (0.5 \times (1.0 - \text{genreAffinity})) + \sum_{i \in \text{Recipe } r} |W_i^-| + \sum_{j \in \text{Genre } G} (0.3 \times |W_j^-|)$$ $$\text{Preference Score} = E[\theta \mid \text{data}] = \frac{\alpha_{\text{post}}}{\alpha_{\text{post}} + \beta_{\text{post}}}$$

Smart Ingredient Substitutions

When a recipe lacks required ingredients, the system executes fuzzy string-matching routines on the ingredient_substitutions seeds to find viable, available alternatives.

Fuzzy Match Logic

The system checks:
1. If $O$ (original name in table) is contained in $M$ (missing recipe ingredient).
2. If $M$ is contained in $O$.
3. If any individual word in $M$ of length $>2$ is contained in $O$.

For multi-part substitutions (e.g. substituting heavy cream with milk + butter), the system parses the ingredients string and evaluates whether all split parts are present in the user's current pantry list. The returned list of matched alternatives is sorted by descending confidence.

Data Metrics & Regression Testing

Every recommendation generates a row in recommendation_logs. Users' interactions dynamically update the user_feedback state from ignored to accepted or cooked.

Using these signals, the MetricsService aggregates logged values to evaluate recommendation performance:

Core Operational Formulas

Let $TP$ (True Positives) = sum of accepted and cooked recommendations, $FP$ (False Positives) = sum of rejected recommendations, and $FN$ (False Negatives) = sum of ignored recommendations:

$$\text{Precision} = \frac{TP}{TP + FP}$$ $$\text{Recall} = \frac{TP}{TP + FN}$$ $$\text{F1-Score} = 2 \cdot \frac{\text{Precision} \cdot \text{Recall}}{\text{Precision} + \text{Recall}}$$ $$\text{Precision@K} = \frac{1}{|U|} \sum_{u \in U} \frac{|\text{Top-K recommendations for user } u \text{ with positive interaction}|}{K}$$

Predefined Scenario Evaluation Harness: To benchmark the scoring weights over modifications, EvaluationService.kt runs a 30-scenario simulation engine. It creates a dummy profile, simulates queries across match profiles (High, Medium, Low, Missing ingredients, Substitutions), simulates feedback based on match score criteria, and spits out mean, median, min, max, and standard deviation summary metrics to the admin panel.
